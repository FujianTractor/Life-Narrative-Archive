# Life-Narrative-Archive 协作开发清单

## 1. 智能体划分方案

本次改造按“分析 -> 拆解 -> 开发 -> 测试 -> 集成 -> 文档整理”推进，采用多智能体协作。

### 智能体 A：总控 / 需求与架构协调

- 负责范围：
  - 对齐既定修改计划
  - 拆分阶段目标与执行顺序
  - 控制改动范围，避免无关扰动
  - 汇总各智能体产出并决定合并顺序
- 输入：
  - `修改计划.md`
  - 当前仓库代码
  - 各智能体的阶段性结果
- 输出：
  - 总体实施路线
  - 阶段验收标准
  - 每轮修改说明
- 依赖关系：
  - 无上游依赖，是协调节点
- 交付物：
  - `开发清单.md`
  - 阶段执行记录
  - 最终变更说明

### 智能体 B：前端迁移与交互设计

- 负责范围：
  - 原生前端迁移到 `Vue 3 + Vite + TypeScript`
  - 页面路由、组件拆分、状态管理、API 封装
  - 关键交互与视觉实现
  - 必要时参考成熟产品或优秀案例，提取可借鉴布局与交互
- 输入：
  - 当前 `index.html`、`script.js`、`styles.css`
  - API 约定
  - 总控智能体的前端阶段任务
- 输出：
  - Vue 工程骨架
  - 页面 / 组件 / Store / Service 实现
  - 前端验证说明
- 依赖关系：
  - 依赖后端接口约定
  - 与集成联调智能体协作
- 交付物：
  - `frontend/` 工程
  - 关键组件与页面
  - 前端联调结果

### 智能体 C：后端重构与 AI / RAG 基础设施

- 负责范围：
  - 从当前 Python / Node 原型迁移到 `Spring Boot`
  - 建立 `Spring AI + LangChain4j + FAISS` 基础设施
  - 建立认证、档案、素材、任务、RAG 等核心后端模块
- 输入：
  - 当前 `main.py` / `server.js`
  - 修改计划中的后端目标结构
- 输出：
  - `backend/` Spring Boot 工程骨架
  - 核心实体、接口、服务、配置
  - RAG / 向量层最小可用实现方案
- 依赖关系：
  - 依赖总控定义阶段边界
  - 为前端和测试智能体提供接口基础
- 交付物：
  - `backend/` 工程
  - API 草案与实现
  - 数据模型草案

### 智能体 D：数据与接口治理

- 负责范围：
  - 梳理数据模型
  - 设计结构化库与 FAISS 的边界
  - 设计 DTO、接口协议、错误返回格式
  - 设计迁移路径（从 `elders.json` / 旧结构过渡）
- 输入：
  - 当前 JSON / SQLite / 上传目录结构
  - 后端模块设计
- 输出：
  - 数据模型清单
  - API 协议约束
  - 迁移策略建议
- 依赖关系：
  - 依赖后端架构方向
  - 输出被后端和测试智能体复用
- 交付物：
  - 数据表设计
  - 接口清单
  - 兼容性说明

### 智能体 E：测试验证

- 负责范围：
  - 为每轮实现提供最小可执行验证
  - 验证基本功能、回归点、关键交互、接口联调
  - 维护 push 前门禁标准
- 输入：
  - 当前阶段已实现代码
  - 功能清单与验收项
- 输出：
  - 测试项清单
  - 测试执行结果
  - 未通过项与风险说明
- 依赖关系：
  - 依赖前端 / 后端已有实现
- 交付物：
  - 测试脚本或执行说明
  - 回归检查结果
  - 发布门禁结论

### 智能体 F：集成联调

- 负责范围：
  - 前后端联调
  - 上传链路联调
  - RAG 请求与来源展示联调
  - 环境变量、启动方式、开发代理配置梳理
- 输入：
  - 已实现的前后端模块
  - 测试结果
- 输出：
  - 联调修复项
  - 集成风险清单
  - 可运行联调版本
- 依赖关系：
  - 依赖前端、后端、测试阶段完成基础能力
- 交付物：
  - 联调通过记录
  - 集成问题清单

### 智能体 G：文档整理与交付说明

- 负责范围：
  - 维护开发清单、实施记录、接口说明、启动说明
  - 每轮修改完成后同步更新文档
- 输入：
  - 各智能体阶段产出
- 输出：
  - README 更新项
  - 模块说明
  - 风险与待办文档
- 依赖关系：
  - 依赖各实施阶段结果
- 交付物：
  - 项目文档
  - 变更摘要
  - 后续待办列表

## 2. 结构化开发清单

## 2.1 功能开发项

### F-01 工程重构骨架

- 新建 `frontend/` 与 `backend/` 目录
- 冻结旧版 `index.html` / `script.js` / `main.py` / `server.js` 作为参考实现
- 建立前后端分离开发结构

### F-02 用户认证能力迁移

- 前端登录 / 注册页面迁移到 Vue
- 后端建立 Spring Security + JWT 基础能力
- 衔接 `auth/me` 保持登录态

### F-03 档案管理能力迁移

- 档案列表、档案详情、档案创建表单迁移

- 上传方式从 `base64 JSON` 改为 `multipart/form-data`

- 统一 chunk 构建规则
- embedding 生成
- FAISS 索引写入与检索
- 问答接口返回来源引用

### F-07 联调与运行方式固化

- 前端代理配置
- 后端配置模板

## 2.7 风险点与注意事项

### R-01 FAISS 不是业务主数据库

- 必须明确其仅承担向量检索角色
- 结构化数据仍需独立存储

### R-02 双 AI 框架职责重复风险

- Spring AI 与 LangChain4j 只能分层协作，不能并行重复实现完整链路

### R-03 当前仓库中文编码风险

- 必须统一为 UTF-8
- 导入旧文件时要注意乱码修复

### R-04 上传链路复杂度

- 视频文件较大
- 后续必然引入异步处理与状态管理

### R-05 改造跨度大

- 这是一次架构迁移，不是局部补丁
- 必须坚持分阶段、小步推进

## 3. 执行顺序

### 阶段 A：分析与拆解

- 复核当前仓库
- 固化协作角色
- 产出开发清单
- 确定阶段边界

### 阶段 B：第一轮开发

- 建立 `frontend/` 骨架
- 建立 `backend/` Spring Boot 骨架
- 不立即大规模迁移旧逻辑，只先搭基础结构

### 阶段 C：功能模块逐步迁移

- 先认证
- 再档案列表与详情
- 再上传链路

### 阶段 D：测试与联调

- 每完成一个模块即执行对应验证
- 每轮保留回归检查清单
- 测试通过后才进入下一轮合并

### 阶段 E：文档整理

- 同步更新开发与启动文档
- 记录风险和未完成项

## 4. 测试与提交策略

### 4.1 测试策略

- 每完成一个功能模块，先做该模块的最小验证
- 至少覆盖：
  - 基本功能可用性验证
  - 相关模块回归检查
  - 必要时接口联调验证
  - 前端关键交互验证
- 如果当前缺少自动化测试框架，先补最基础可执行验证方式

### 4.2 提交策略

- 小步修改，小步验证
- 不在核心功能未验证通过前执行 push
- 一轮只解决一类问题，避免跨模块大杂烩式提交
- 先本地验证，再考虑 git 提交与 push

### 4.3 当前实施原则

- 先阅读上下文，再修改
- 尽量减少对无关模块的扰动
- 关键逻辑补必要注释
- 保持风格一致
- 每轮结束输出：
  - 本轮修改内容
  - 影响范围
  - 已验证项
  - 下一轮待办
