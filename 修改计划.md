# Life-Narrative-Archive 扩充改造计划（Spring AI + LangChain4j + FAISS 版）

## 1. 目标调整

本次项目扩充采用以下目标技术路线：

- 前端：`Vue 3 + Vite + TypeScript`
- 后端：`Spring Boot`
- AI 编排：`Spring AI + LangChain4j`
- RAG：基于文档切片、向量检索、上下文增强回答
- 向量库：`FAISS`
- 业务数据存储：建议保留一个关系型数据库存档案、用户、素材元数据

这里先明确一个关键点：

`FAISS` 更准确的定位是“向量索引 / 向量检索引擎”，不适合作为整个系统唯一的业务数据库。也就是说：

1. `FAISS` 负责存和查 embedding 向量。
2. 用户、档案、时间线、素材、任务状态这些结构化数据，仍建议使用 `MySQL` / `PostgreSQL` / `SQLite` 中的一种。

如果你坚持“数据库使用 FAISS”，从工程上可以理解为“向量检索层使用 FAISS”，而不是“所有业务数据都只放进 FAISS”。这部分我已经在本计划里按可实施方案处理。

## 2. 结合当前仓库的现状判断

当前仓库实际情况：

- 前端仍是原生静态页：`index.html`、`script.js`、`styles.css`
- 后端同时有：
  - `main.py`：FastAPI 版本
  - `server.js`：Node 原型版本
- 当前数据层是混合方案：
  - `data/app.db`
  - `data/elders.json`
  - `uploads/`
  - `data/rag/`
- 当前 RAG 是简化版：
  - `LangChain + FAISS`
  - embedding 仍是本地简化实现
  - 图片 / 视频没有真正做 OCR、ASR、视觉理解

这意味着如果要切换到你指定的新架构，不是“小改”，而是一次明确的后端重构。前端 Vue 化和后端 Java 化可以并行推进，但建议分阶段实施。

## 3.3 AI 层职责划分

由于你指定的是 `Spring AI + LangChain4j` 双栈，建议明确分工，避免两套抽象互相重叠：

### Spring AI 负责

- 模型配置与接入管理
- Embedding 生成
- Prompt / ChatModel 调用封装
- 向量检索流程中的 Spring 集成
- 与 Spring Boot 配置体系深度结合

### LangChain4j 负责

- AI Service 风格接口定义
- 结构化输出与工具调用封装
- 对话式问答服务抽象
- 后续多轮问答、摘要、信息抽取等服务接口定义

### 建议原则

- 以 `Spring Boot` 作为统一工程框架。
- 以 `Spring AI` 作为模型与 embedding 主接入层。
- 以 `LangChain4j` 作为 AI 服务封装层。
- 不要在两个框架里重复实现一套完整 RAG 链路，否则维护成本会很高。

## 6.2 FAISS 存储定位

`FAISS` 负责：

- 存放向量索引
- 执行相似度检索
- 支持按档案重建或增量重建索引

建议存储方式：

- 每个档案一个 FAISS 索引目录
- 或全局一个索引 + metadata 映射表

两种方案比较：

### 方案 A：每个档案一个 FAISS 索引

优点：

- 管理简单
- 删除、重建某个档案成本低
- 易于隔离数据

缺点：

- 跨档案检索不方便
- 档案数增多后索引文件管理会复杂

### 方案 B：全局一个 FAISS 索引

优点：

- 统一管理
- 支持跨档案检索和推荐

缺点：

- 删除 / 更新单个档案更复杂
- 需要更强的 metadata 映射管理

对当前项目，建议优先使用：

- 第一阶段：`每个档案一个 FAISS 索引`
- 第二阶段：视数据规模再升级为全局索引

## 7. 多模态 RAG 设计

## 7.1 第一阶段的正确实现方式

不要一开始追求真正的跨模态 embedding 全覆盖，建议先做“多模态内容文本化后的 RAG”：

1. 用户上传文字、图片、视频。
2. 系统对不同模态做内容提取。
3. 把提取结果统一转成结构化文本。
4. 文本切片后生成 embedding。
5. embedding 写入 FAISS。
6. 查询时先检索，再用大模型回答。

这条路线最稳，也最容易在 `Spring AI + LangChain4j + FAISS` 架构里落地。

## 7.2 各模态处理方式

### 文本

- 清洗无效字符
- 按段落 / 字数切片
- 生成 embedding
- 写入 FAISS

### 图片

- OCR 提取图中文字
- Vision 模型生成图片描述
- 可选：抽取人物、地点、事件标签
- 合并为文本知识片段
- 切片后写入 FAISS

### 视频

- FFmpeg 抽取音轨
- ASR 生成转写文本
- 定时抽取关键帧
- 对关键帧调用 Vision 模型生成描述
- 可选：生成整段视频摘要
- 合并为文本知识片段
- 切片后写入 FAISS

## 7.3 RAG 查询链路

建议链路如下：

```text
用户提问
-> Query 标准化
-> Embedding 生成
-> FAISS TopK 检索
-> 读取 chunk 元数据
-> 组装上下文
-> Spring AI / LangChain4j 调用大模型生成回答
-> 返回答案 + 引用来源
```

## 7.4 回答输出建议

每次问答结果建议返回：

- `answer`
- `sources`
- `sourceType`
- `archiveId`
- `archiveName`
- `workflow`

例如：

- 来源是时间线
- 来源是图片 OCR
- 来源是视频转写
- 来源是视频关键帧描述

这样前端可以把“可解释性”做出来。

## 8. Spring AI 与 LangChain4j 的集成建议

## 8.1 建议的调用层次

建议用下面这种层次，而不是混写：

```text
Controller
-> RagService
-> RetrieverService (FAISS 检索)
-> PromptAssembler
-> ChatService
   -> Spring AI ChatModel / EmbeddingModel
   -> LangChain4j AiService
```

## 8.2 适合 Spring AI 的部分

- 模型配置放在 `application.yml`
- embedding 模型统一管理
- ChatModel 统一注入
- 与 Spring Boot 配置、环境变量整合

## 8.3 适合 LangChain4j 的部分

- 定义 `@AiService` 风格的问答接口
- 定义结构化信息抽取接口
- 为后续“摘要生成”“标签抽取”“多轮问答”提供统一服务接口

## 8.4 集成注意点

注意避免这几种重复建设：

1. 同时在 Spring AI 和 LangChain4j 中各写一套 embedding 调用
2. 同时在两个框架里各写一套 prompt 拼装
3. 同时维护两套文档切片与检索逻辑

建议：

- embedding、检索、chunk 管理只保留一套实现
- LangChain4j 主要做 AI service 抽象
- Spring AI 主要做模型与 Spring 集成

## 12. 优先级建议

为了减少返工，建议按下面顺序推进：

1. 新建 `frontend/` 与 `backend/` 工程骨架
2. 先迁 Vue 前端
3. 再建 Spring Boot 的基础 CRUD 和认证
4. 再改上传链路为 `multipart/form-data`
5. 再接文本切片与 FAISS 检索
6. 再接图片 OCR / 视频 ASR
7. 最后补齐 Spring AI + LangChain4j 问答层

原因：

- 先把基础工程搭起来，后面功能迭代才稳
- 如果一开始直接冲多模态 RAG，会被上传、索引、元数据管理拖慢
- 先实现“文本 RAG 可跑通”，再逐步把图片和视频接进来，风险更低

## 13. 风险与注意事项

## 13.1 当前仓库中文编码风险

当前代码和输出中已经有中文乱码痕迹。迁移时必须统一：

- 源码文件编码统一 `UTF-8`
- 数据导出导入统一 `UTF-8`
- 前后端响应统一 `UTF-8`

## 13.2 Java 侧使用 FAISS 的工程复杂度

需要明确：

- `FAISS` 在 Python 生态中接入更直接
- 在 Java / Spring 体系中，通常需要额外封装
- 这意味着你选 `Spring Boot + FAISS` 是可行的，但实现复杂度会高于“直接选 pgvector / Milvus / Elasticsearch 向量检索”

所以建议：

- 第一阶段先把 FAISS 作为独立索引管理模块处理
- 不要让业务代码到处直接耦合 FAISS 细节

## 13.3 不建议继续沿用 elders.json 作为主数据源

随着图片、视频、转写、OCR、关键帧、chunk 增多，JSON 会迅速失控。

必须尽快迁到数据库表结构。

## 13.4 双 AI 框架的重复建设风险

`Spring AI` 和 `LangChain4j` 能一起用，但前提是职责必须清楚。

如果不分工，会出现：

- 两套 prompt
- 两套模型调用
- 两套 embedding 封装
- 两套 RAG 逻辑

这会让维护成本明显上升。

## 14. 最终建议

建议把项目拆成两个里程碑：

### 里程碑 A：Spring 化与 Vue 化

- Vue 前端完成
- Spring Boot 基础 API 完成
- 用户、档案、素材改为数据库管理
- 文件上传链路完成

### 里程碑 B：多模态 RAG 完成

- 文本 chunk 与 embedding 完成
- FAISS 检索层完成
- 图片 OCR / 图片描述接入
- 视频 ASR / 抽帧 / 摘要接入
- Spring AI + LangChain4j 问答链路完成
- 前端展示答案与来源引用

按这个方案推进，项目会从当前原型升级为：

- 前端：Vue 组件化工作台
- 后端：Spring Boot 业务服务
- AI 层：Spring AI + LangChain4j
- 检索层：FAISS
- 能力层：多模态资料入库、知识抽取、RAG 问答
